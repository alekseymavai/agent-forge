## 3. Gift Protocol

Gift — единица передачи данных между ролями. Каждая роль получает Gift
от предыдущей и создаёт новый Gift для следующей.

### Структура Gift

```python
@dataclass
class Gift:
    content: dict       # данные от роли
    freedom: Freedom    # ACCEPTED / DEFERRED / DECLINED
    source_role: str    # кто создал
    target_role: str    # кому предназначен
    timestamp: str      # когда создан
```

### Freedom (свобода выбора)

| Значение | Смысл | Кто может |
|----------|-------|-----------|
| ACCEPTED | Задача принята, работа выполнена | Все роли |
| DEFERRED | Отложено / возврат на доработку | Хозяин, Испытатель |
| DECLINED | Отклонено / заблокировано | Блюститель, Наставник |

| **Анамнезис** | Память | Patterns, Lessons, ADR — команда помнит |
| **Телос** | Цель за пределами задачи | Продукт полезен людям, команда растёт |
| **Свобода** | Право выбора | ACCEPTED / DEFERRED / DECLINED |

## 5. Консенсус

Решения команды принимаются через взвешенное голосование.

### Формула

```
score = Σ (weight_i × vote_i) / Σ weight_i
```

Где `vote_i` ∈ {+1 (ACCEPTED), 0 (DEFERRED), -1 (DECLINED)}.

### Блокировка

- **Блюститель** (1.3): DECLINED при HIGH-уязвимости блокирует пайплайн
- **Наставник** (1.5): DECLINED останавливает всё

### Финальное решение

```
ConsensusReport.human_decision_required = True  # ВСЕГДА
```

Команда предлагает — человек решает. Без исключений.

### Антипаттерн

> Код без session-log в коммите — контекст решений теряется.
> Session-log обновляется В ТОМ ЖЕ коммите что и код.

## 7. Возвраты и циклы

### Блюститель → Зодчий

При HIGH-уязвимости: Блюститель блокирует, задача возвращается Зодчему
на переработку плана.

### Испытатель → Делатели

При красных тестах: Испытатель возвращает задачу Делателям
с описанием проблем. Сам не исправляет.

### Хозяин → Наставник (DEFER)

При несоответствии требованиям: Хозяин откладывает задачу,
Наставник фиксирует причину в памяти.

---

## 8. Параллельная работа

Делатель-тыл (BackendDev) и Делатель-лик (FrontendDev) работают параллельно:

- Оба получают план от Зодчего и вердикт Блюстителя
- Работают независимо по своим файлам
- Результаты сливаются перед Испытателем
- API-контракты согласованы заранее через план Зодчего

---

## 9. Верификация

Урок команды: субагент может отчитаться об успехе, но файлы не изменятся.

**Правило**: после каждого Write/Edit — верифицировать через grep/cat.
Координатор после subagent DONE — независимая проверка `python3 -m pytest`.

## 10. Безопасность

- Токены изолированы по воркспейсам (ADR-003)
- Публичные эндпоинты: только `bool(error)`, детали — за аутентификацией
- Секреты: только через `os.environ[KEY]` без fallback
- Security Review обязателен перед любым деплоем
