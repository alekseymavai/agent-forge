## Проблема

Каждая сессия — амнезия. Агент начинает заново. Архитектурные решения теряются.
Антипаттерны повторяются. Опыт не накапливается.

```
Сейчас:   сессия 1 → опыт → сессия 2 → опыт теряется → сессия 3 → снова с нуля
Цель:     сессия 1 → опыт → TeamMemory → сессия 2 → анамнезис → сессия N → мастерство
```

---

## Три аксиомы AgentForge

### 1. Роль — лицо с призванием (не функция)

Scout не "собирает данные". Scout **дарит карту** Architect-у — вкладывая в неё своё призвание.
Каждая роль знает:
- **Логос** — свою природу (что я есть)
- **Телос** — свою цель (зачем я здесь)
- **Анамнезис** — что было до меня

### 2. Агент — слуга, не господин

Агент анализирует, предлагает, предупреждает.
Финальное решение — **всегда за человеком** (`human_decision_required: True`).

```python
return ConsensusReport(
    gifts=[scout_gift, architect_gift, security_gift],
    recommendation="вариант B — меньше риска",
    human_decision_required=True,
    note="Решение принимает Андрей."
)
```

### 3. Память — анамнезис, не архив

Завершённая задача (✅) не закрыта — она **со-присутствует** в следующей.
`Obsolete` в PLM — не смерть, а передача опыта.

```
M.5 решена → TeamMemory(devteam) → M.6 знает как решали M.5
```

## Gift Protocol

Каждая передача эстафеты между ролями — **дар**:

```python
@dataclass
class Gift:
    giver: str       # кто дарит ("scout")
    receiver: str    # кому ("architect")
    content: Any     # содержание (ScoutReport, Plan, ...)
    telos: str       # зачем — связь с телосом задачи
    anamnesis: list  # что помним из прошлого
    freedom: str     # ACCEPTED | DEFERRED | DECLINED
    cost: int        # кеносис — стоимость дара (токены, время)
```

**A5 — Свобода:** получатель может ответить `DEFERRED`.
Дар откладывается, не теряется. Coordinator ждёт или запрашивает другой дар.

## Команда = Завод

Завод — не конвейер функций. Завод — **сообщество лиц**, каждое из которых:
- знает своё место в жизненном цикле продукта
- дарит результат следующему
- помнит всё что было до

### Жизненный цикл задачи (из PLM-GIFT)

```
ТЕЛОС (Андрей ставит задачу)
  ↓
In Design  → [Scout] → [Architect] → выбор плана
  ↓
In Review  → [QA] → [Security] → challenge
  ↓
Released   → [DevOps] деплой → [TechWriter] документация
  ↓
Obsolete   → живёт в TeamMemory как анамнезис следующей задачи
```

### CRM как PLM-модуль

CRM — не "где хранятся заказы". CRM — **модуль управления жизненным циклом отношений с клиентом**:

```
Клиент    = лицо с анамнезисом всех взаимодействий
Жалоба    = ECR (рана)
Решение   = ECO (дар исцеления)
Доставлен = Released
Неактивен = Obsolete (живёт в памяти, не удалён)
```

## Иерархия телоса

При конфликте — всегда побеждает более высокий уровень:

```
Телос задачи (польза для пользователя)
  > Security  (нет уязвимостей) × 1.3
    > QA       (тесты зелёные)  × 1.2
      > Architect (план согласован) × 1.0
        > простота кода
          > скорость
```

## Code Memory Graph

Не плоские таблицы — **живой граф анамнезиса**:

```
(задача:M5) -[ПОРОДИЛА]→  (паттерн:AgentContext)
(задача:M6) -[ИСПОЛЬЗУЕТ]→ (паттерн:AgentContext)
(задача:M6) -[ПОМНИТ]→    (задача:M5)
(паттерн:AgentContext) -[В_ФАЙЛЕ]→ (файл:memory_service.py)
```

Хранится в Integram `devteam` через Neo4j.
Каждый агент может запросить: "что было создано похожего на эту задачу?"
